Codex 是如何在长任务里保持连续性的？

关于 compact()、handoff summary 与上下文压缩的一篇整理

我这里先做一个中文整理版。由于 X 的 article 页面当前不太好直接抓取全文，下面的内容是根据原始链接及公开转述整理出来的核心要点，重点讨论 Codex 在长上下文任务里是如何保持连续性的。

1. 问题是什么？ ¶

很多人在实际使用 Codex 的时候都会有一个直观感受：

  • 它在长任务里比很多普通聊天式模型更“不断片”
  • 即使任务持续很久，它也常常还能维持比较稳定的目标感
  • 它不像简单截断上下文那样，突然把前面做过的事忘掉

这篇文章想解释的，就是这种连续性背后一个很关键的机制：compact()

2. 不是简单丢历史，而是做一次“交接” ¶

直觉上，当上下文太长时，最粗暴的做法是：

  • 丢掉最早的一段对话
  • 或者简单截断 token

但这样做的问题很明显：模型会失去任务背景、设计决策、已完成步骤，以及很多隐含约束。

文章里描述的 Codex 做法不是“直接丢”，而是先做一次压缩交接。

当上下文增长到一定程度时，系统会触发一次 compact()，把长历史浓缩成一份更短、但尽量保留任务状态的 handoff summary。

3. 文章给出的核心流程 ¶

根据文中的逆向分析，这个流程大致可以概括成下面几步：

  1. 当会话历史过长时，客户端不会继续把完整历史原样塞给主模型，而是触发 compact()
  2. 服务端会调用一个专门负责压缩上下文的模型，把整个会话历史整理成一份 handoff summary
  3. 这份 summary 不会明文直接发回客户端，而是被加密封装成一个 blob
  4. 下一次真正调用 responses.create() 时，客户端把这个 blob 一起带回服务端
  5. 服务端再把它解开，并把压缩后的任务状态与新的上下文一起喂给主模型

也就是说，系统并不是让主模型每一轮都“重新看完全部历史”，而是通过一次中间交接，把长历史变成了更短、更结构化的任务状态。

4. 为什么这种方式有效？ ¶

这种机制有效，主要有几个原因。

4.1 它保留的是“任务状态”，不是字面历史 ¶

在很多真实任务里，真正重要的并不是逐字逐句保留所有聊天记录，而是保留：

  • 当前目标是什么
  • 已经做了哪些尝试
  • 哪些方案已经失败
  • 当前代码库或文件状态是什么
  • 下一步计划是什么
  • 有哪些不能违反的约束

如果这些内容能被高质量地抽取出来，那么模型即使不再看到全部原始对话，也依然能保持较强的连续性。

4.2 它减少了主模型重复消费长历史的成本 ¶

如果每一轮都把全部上下文重新发送给主模型，那么：

  • token 消耗会不断膨胀
  • 延迟会升高
  • 长历史里大量低价值内容会持续占据上下文窗口

而压缩之后，主模型看到的是一份更短、更聚焦的 handoff 状态。这会让系统在长任务里更稳定，也更省 token。

4.3 它把“压缩”从客户端挪到了服务端 ¶

这点很重要。

如果是客户端自己做总结，那么用户很难保证总结质量，也很难保证总结格式稳定。
而如果这一步由服务端统一完成，就可以：

  • 使用专门的 compactor 模型
  • 使用统一的总结结构
  • 让主模型接收更标准化的 handoff 输入

从系统设计角度看，这其实更像一种受控的上下文状态管理，而不是简单的“聊天摘要”。

5. 这解释了什么现象？ ¶

这套机制可以解释一些在 Codex 里非常常见的体验：

  • 为什么它做长工程任务时衔接感比较强
  • 为什么它比普通对话产品更像一个“持续工作的 agent”
  • 为什么它在多轮修改、重构、调试里不那么容易完全漂移

当然，这不意味着压缩是无损的。
任何 summary 本质上都是有损压缩，只是这个系统看起来把压缩做得足够工程化，因而在大多数任务里可以接受。

6. 这对做 Agent 有什么启发？ ¶

我觉得这篇文章真正有价值的地方，不只是解释了 Codex，而是给“如何做一个能长期工作的 agent”提供了几个很强的启发：

6.1 上下文管理是核心能力，不是外围细节 ¶

很多 agent 系统的问题，不在模型本身，而在于：

  • 不会管理上下文
  • 不会交接状态
  • 不会把长期任务拆成可持续推进的工作流

如果没有一套像样的 handoff 机制，任务一长就会崩。

6.2 需要区分“原始历史”和“工作记忆” ¶

原始历史适合做审计和回溯。
而工作记忆更应该是：

  • 当前目标
  • 当前状态
  • 已知约束
  • 已完成步骤
  • 未完成事项

后者更像真正能支撑 agent 持续工作的短期记忆结构。

6.3 压缩器本身可以是独立模块 ¶

文章里最有意思的一点，是它暗示了一个架构思路：

  • 主模型负责解决问题
  • 辅助模型负责整理上下文

也就是说，“解决问题的模型” 和 “维护任务状态的模型” 可以不是同一个。

这比让一个模型同时做所有事情，往往更合理。

7. 我的理解 ¶

如果把这篇文章压缩成一句话，那大概就是：

Codex 在长任务里之所以显得连续，不是因为它永远记住了所有历史，而是因为它有一套更成熟的上下文交接机制。

对一个工程型 agent 来说，真正重要的不是“记住一切”，而是“在关键时刻把任务状态正确地交给下一轮”。

这件事一旦做好，系统的表现就会从“多轮聊天”变成“持续协作”。
